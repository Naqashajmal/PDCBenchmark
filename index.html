Producer–Consumer
2 producer threads and 3 consumer threads share a bounded ArrayBlockingQueue. Producers block when the buffer is full; consumers block when it is empty. No manual locking required.

Java — BlockingQueue
BlockingQueue<Integer> q = new ArrayBlockingQueue<>(bufSize);

// Producer — blocks when buffer is full
q.put(item);

// Consumer — blocks when buffer is empty
int item = q.take();

// Poison pill: signals consumers to stop
q.put(-1);
Synchronization & Mutex
Runs the same counter increment across multiple threads three ways: no lock (race condition), ReentrantLock (mutex), and AtomicInteger (lock-free CAS). Shows real data corruption.

Results

Unsafe — — — No synchronization

Java — three approaches
// Unsafe — NOT atomic, threads corrupt each other's writes
volatile int counter = 0;
counter++; // read → modify → write: three steps, not one

// ReentrantLock — only one thread holds the lock at a time
ReentrantLock lock = new ReentrantLock();
lock.lock();
try { counter++; } finally { lock.unlock(); }

// AtomicInteger — lock-free, single hardware instruction
AtomicInteger c = new AtomicInteger(0);
c.incrementAndGet();
